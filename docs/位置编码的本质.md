## 一、为什么需要位置编码？

在前面的章节中，我们已经知道了 Token Embedding 的作用——把每个 token ID 映射成一个 384 维的稠密向量，让"你"、"好"、"吗"这些字在模型内部有了各自的语义坐标。但如果只有 Token Embedding，模型就会遇到一个非常致命的问题：**它分不清语序。**

来看一个例子。两个句子：

```
句子A："狗 咬 人"
句子B："人 咬 狗"
```

它们的 token 完全一样，只是顺序不同。如果只靠 Token Embedding，模型拿到的是什么？

```
句子A "狗咬人":
  token_emb["狗"] = [0.3, 0.7, -0.1, 0.5]
  token_emb["咬"] = [0.8, -0.2, 0.4, 0.1]
  token_emb["人"] = [0.1, 0.6, 0.3, -0.4]

句子B "人咬狗":
  token_emb["人"] = [0.1, 0.6, 0.3, -0.4]   ← 跟句子A一模一样
  token_emb["咬"] = [0.8, -0.2, 0.4, 0.1]   ← 一模一样
  token_emb["狗"] = [0.3, 0.7, -0.1, 0.5]   ← 一模一样
```

进入 Attention 的时候，Q·K 点积本质上是一种**集合运算**——"狗"的 Q 会和所有 token 的 K 做点积。不管"狗"排在位置 0 还是位置 2，它的向量都是 `[0.3, 0.7, -0.1, 0.5]`，算出来的注意力分数完全相同。对模型来说，"狗咬人"和"人咬狗"是等价的。这两个句子，一个是新闻，一个是奇闻，但模型却完全分不清。

**问题的根源在于：Token Embedding 只回答了"是什么"，没有回答"在哪里"。**

### 加上 Position Embedding 之后

Nova 在 Token Embedding 之外，额外维护了一张 Position Embedding 表。在前向传播的最初阶段，每个 token 的输入向量 = Token Embedding + Position Embedding：

```python
# model.py — NovaModel.forward
token_emb = self.token_emb(input_ids)                   # 查字义表
positions = torch.arange(seq_len, device=input_ids.device)  # [0, 1, 2, ..., seq_len-1]
pos_emb   = self.pos_emb(positions)                     # 查位置表
x = token_emb + pos_emb                                # 叠加
```

加上位置信息后，同一个"人"字在不同位置会得到不同的向量：

```
句子A "狗咬人":
  "狗" 的输入 = token_emb["狗"] + pos_emb[0] = [0.3, 0.7, -0.1, 0.5] + [0.9, 0.1, 0.0, -0.2]
                                              = [1.2, 0.8, -0.1, 0.3]
  "咬" 的输入 = token_emb["咬"] + pos_emb[1] = [0.8, -0.2, 0.4, 0.1] + [0.4, 0.5, -0.3, 0.1]
                                              = [1.2, 0.3, 0.1, 0.2]
  "人" 的输入 = token_emb["人"] + pos_emb[2] = [0.1, 0.6, 0.3, -0.4] + [-0.1, 0.2, 0.7, 0.3]
                                              = [0.0, 0.8, 1.0, -0.1]

句子B "人咬狗":
  "人" 的输入 = token_emb["人"] + pos_emb[0] = [0.1, 0.6, 0.3, -0.4] + [0.9, 0.1, 0.0, -0.2]
                                              = [1.0, 0.7, 0.3, -0.6]  ← 跟句子A中的"人"不同了
  "咬" 的输入 = token_emb["咬"] + pos_emb[1] = [1.2, 0.3, 0.1, 0.2]  ← 同位置同token，碰巧一样
  "狗" 的输入 = token_emb["狗"] + pos_emb[2] = [0.3, 0.7, -0.1, 0.5] + [-0.1, 0.2, 0.7, 0.3]
                                              = [0.2, 0.9, 0.6, 0.8]  ← 跟句子A中的"狗"不同了
```

同一个"人"字，在位置 0 和位置 2 加了不同的 pos_emb 后，向量就不一样了。进入 Attention 后，Q·K 点积的结果自然不同，模型终于能区分"狗咬人"和"人咬狗"了。

所以 Position Embedding 的作用，一句话讲完：**给每个 token 贴一个"我在第 N 个位置"的标签，让模型在后续的 Attention 计算中能感知到语序。**

### Token Embedding 和 Position Embedding 的职责划分

| 组件 | 类比 | 回答的问题 | 是否理解上下文 |
|------|------|-----------|--------------|
| Token Embedding | 字典 | 这个字**是什么**（"狗"的字典释义） | 否，静态查表 |
| Position Embedding | 座位号 | 这个字**在第几个位置** | 否，静态查表 |
| token_emb + pos_emb | 贴了座位号的字典卡片 | "是什么字" + "在哪个位置" | 否，仍然是静态的 |
| **Transformer Block** | **大脑** | **综合所有卡片，理解上下文含义** | **是，语义理解从这里才真正开始** |

"下一个 token 是什么"这个能力，100% 来自 Transformer Block 中的自注意力计算和前馈网络，不在 Token Embedding 里，也不在 Position Embedding 里。这两张表只是给 Block 提供最原始的输入信息：一个告诉它"这个字是什么"，一个告诉它"这个字在哪"。没有 Position Embedding，Block 就是一个分不清语序的大脑——能认字，但不知道字的排列顺序。

### pos_emb 表只有 128 行，怎么够用？

Nova 的 Position Embedding 表定义如下：

```python
# model.py — NovaModel.__init__
self.pos_emb = nn.Embedding(config.max_seq_len, config.d_model)
# 等价于 nn.Embedding(128, 384)，一张 128 行 × 384 列的表
```

128 行，每行 384 维。而 Nova 的训练数据有 4 亿 token，按 128 长度切分大约是 **312 万条序列**。一个直觉性的疑问是：312 万条句子的第一个 token 都不一样，pos_emb[0] 只有一个 384 维向量，怎么放得下 312 万种不同的位置语义？

答案是：**它不需要放 312 万种语义。它只放一种语义："这是第 0 个位置。"**

这里容易出现的思维误区是，把 pos_emb 想成了"每个句子的每个位置都需要一套独立的位置信息"。但实际上，pos_emb[0] 根本不关心这个位置上站的是"我"、"天气"还是"你"，它存的东西跟具体 token 毫无关系：

```
312万条句子的第一个token：

第1条句子：    "我"   + pos_emb[0]
第2条句子：    "天气" + pos_emb[0]    ←  完全相同的 pos_emb[0]
第3条句子：    "你"   + pos_emb[0]    ←  完全相同的 pos_emb[0]
...
第312万条：    "今天" + pos_emb[0]    ←  完全相同的 pos_emb[0]
```

312 万次，加的都是一模一样的 384 个数字。那区分不同句子靠什么？靠 Token Embedding。正是因为每条句子的第一个 token 不一样，它们查出来的 token_emb 不一样，所以最终叠加后的输入向量也不一样：

```
句子1第一个位置：  token_emb["我"]   + pos_emb[0] = [-0.3, 0.5, ...] + [0.1, 0.2, ...] = [-0.2, 0.7, ...]
句子2第一个位置：  token_emb["天气"] + pos_emb[0] = [ 0.8, -0.1, ...] + [0.1, 0.2, ...] = [ 0.9, 0.1, ...]
```

结果不同，是因为 token_emb 不同，不是因为 pos_emb 不同。

打一个最简单的比方：把 pos_emb 想成**邮政编码**。你住北京朝阳区，邮编是 100020；你邻居也住朝阳区，邮编也是 100020；全朝阳区 300 万人，邮编都是 100020。邮编不需要"装得下 300 万人的信息"，它只表达一件事："这个地方是朝阳区。"至于住的是谁，那是身份证的事，不是邮编的事。

```
人1 = 身份证(张三) + 邮编(100020)   →  "住在朝阳区的张三"
人2 = 身份证(李四) + 邮编(100020)   →  "住在朝阳区的李四"
人3 = 身份证(王五) + 邮编(100025)   →  "住在丰台区的王五"
```

同理：

```
token1 = token_emb(我)   + pos_emb[0]  →  "在位置0的'我'"
token2 = token_emb(天气) + pos_emb[0]  →  "在位置0的'天气'"
token3 = token_emb(天气) + pos_emb[3]  →  "在位置3的'天气'"
```

**pos_emb[0] 永远只说一件事："这里是位置 0。"** 不多，不少。所以 128 行完全够用——因为一个序列最多 128 个位置，128 行正好覆盖每一个位置。

### 位置向量需要训练吗？

理解到这里，另一个自然而然的疑问就来了：如果 pos_emb 仅仅只是表示"这里是位置 0"、"这里是位置 1"，那为什么还要在反向传播中花力气去训练它？这有什么好优化的？

这个直觉其实相当准确。但问题在于，**模型不认识"数字"，它只认识"向量"**。人类天然理解 0、1、2、3 的顺序关系，但模型的整个计算流程——Q·K 点积、softmax、FFN 矩阵乘法——全部都是向量间的数学运算。你没法把一个整数 `0` 直接丢进去做矩阵乘法，你必须把"位置 0"翻译成一个 384 维的向量，模型才能用。

那这 384 个数字应该填什么？如果随便填，比如用 one-hot 编码：

```
pos_emb[0] = [1, 0, 0, 0]    ← "第0个位置"
pos_emb[1] = [0, 1, 0, 0]    ← "第1个位置"
pos_emb[2] = [0, 0, 1, 0]    ← "第2个位置"
pos_emb[3] = [0, 0, 0, 1]    ← "第3个位置"
```

这样确实能区分位置，但位置 0 和位置 1 的向量距离，跟位置 0 和位置 3 的距离是一样的（都是 √2）。模型完全感知不到"位置 1 离位置 0 更近"这件事。一个好的位置向量，至少需要满足：相邻位置的向量应该相似、不同位置必须可区分、要跟后续 Q·K 点积配合良好——这些要求非常复杂，人类设计不出最优的 384 维向量，所以在 GPT-2 的方案中，就让模型自己通过训练来学习。

但有趣的是，这个直觉在学术界的演进中也得到了验证：

**2017 年**，Vaswani 的原始 Transformer 论文就觉得"位置就是位置，没什么好学的"，直接用正弦/余弦公式把位置向量算死，不参与训练。

**2018～2020 年**（GPT 时代），有人试了可学习的 pos_emb，发现效果略好一点，因为模型能学到一些不那么均匀的位置模式（比如第 0 个位置比较特殊，因为经常是 `<s>` 起始符）。于是流行了一阵。Nova 当前用的就是这个方案。

**2021 年至今**（LLaMA 时代），业界又回到了"不训练"的路线——RoPE 旋转位置编码，用数学公式直接生成，零训练参数。而且效果比学习的还好，还支持外推到更长的上下文。

也就是说，位置编码确实不需要太多训练，这个判断是对的。Nova 的 pos_emb 只占总参数量的 1.24%（49,152 / 1,322,112），在整个模型中的存在感很低。

## 二、可学习绝对位置嵌入的弊端

理解了 Position Embedding 的本质之后，接下来要回答的问题是：Nova 当前使用的这套位置编码方案，到底有什么问题？

### 本质：一张查找表，表有多大，上下文就有多长

Nova 的 Position Embedding 是一张 `[128, 384]` 的查找表，128 行对应 128 个位置（位置 0 到位置 127）。前向传播时，模型生成 `[0, 1, 2, ..., seq_len-1]` 的位置索引，然后拿着索引去查表：

```python
# model.py — NovaModel.__init__
self.pos_emb = nn.Embedding(config.max_seq_len, config.d_model)
# config.max_seq_len = 128，所以表是 128 行 × 384 列

# model.py — NovaModel.forward
positions = torch.arange(seq_len, device=input_ids.device)  # [0, 1, 2, ..., seq_len-1]
pos_emb = self.pos_emb(positions)                            # 查表
```

这里的 128 不是算出来的，它就是 config 里手动设定的超参数 `max_seq_len`。你写 128 就是 128 行，写 256 就是 256 行，写 2048 就是 2048 行。它跟词表大小（vocab_size = 16000）无关，跟训练数据总量（4 亿 token）也无关，纯粹取决于你希望模型一次能处理多长的句子。

### 致命缺陷：表没有的行，查不了

问题来了。训练时 `max_seq_len = 128`，表只有 128 行。如果推理时输入超过 128 个 token，位置索引就会出现 128、129、130……这些索引在表里根本没有对应的行，直接越界报错。

所以在推理代码中，不得不做一个硬截断：

```python
# chat.py — generate 函数
ids_cond = ids[:, -model.pos_emb.weight.shape[0]:]  # 超出 128 就截掉前面的
```

这意味着，如果用户的对话越聊越长，超过 128 个 token 后，模型只能看到最近的 128 个 token，更早的上下文会被直接丢弃。对于一个对话模型来说，这相当于"聊着聊着就失忆了"。

### 为什么不能直接把表扩大？

一个朴素的想法是：训练时直接把 `max_seq_len` 设成 8192 甚至更大，表不就够用了吗？

理论上可以，但代价很高：

| max_seq_len | Position Embedding 参数量 | Attention 计算复杂度 |
|-------------|--------------------------|---------------------|
| 128 | 128 × 384 = 49K | O(128²) |
| 2048 | 2048 × 384 = 786K | O(2048²) = 256 倍 |
| 8192 | 8192 × 384 = 3.1M | O(8192²) = 4096 倍 |

Position Embedding 参数量的增长还算线性，真正致命的是 Attention 的计算开销——自注意力的 QK 矩阵计算复杂度是 O(n²)，n 是序列长度。把上下文从 128 扩到 8192，Attention 的计算量直接暴涨 4096 倍。对于 Nova 这样的微型模型来说，在资源有限的单卡上这样做几乎是不现实的。

更根本的问题在于，即使你付得起训练成本把表扩到了 8192，推理时仍然不能超过 8192。这就是可学习绝对位置嵌入的根本局限性：**训练时用多长的上下文，推理时就只能用多长。** 它的上下文窗口被一张固定大小的查找表锁死了，无法外推。

### 那业界怎么做到 128K 上下文的？

答案是：**换一套位置编码方案。** 目前主流的大模型（LLaMA、Qwen、DeepSeek 等）全部使用 RoPE（Rotary Position Embedding，旋转位置编码）。RoPE 不维护任何查找表，而是在 Attention 内部用数学公式对 Q 和 K 向量做旋转，通过旋转角度来编码位置信息。由于公式对任意位置都能计算，配合位置插值等技术，就可以在训练时只用 8K 上下文，推理时扩展到 32K、64K 乃至 128K。

| | 可学习绝对位置嵌入（Nova 当前） | RoPE（LLaMA 系列） |
|---|---|---|
| 位置信息怎么来的 | 训练学出来的查找表 | 数学公式（旋转角度），不可学习 |
| 位置信息加在哪 | 输入向量上（进 Block **之前**） | Q 和 K 上（Attention **内部**） |
| 编码的是什么 | **绝对位置**（我在第 5 个位置） | **相对距离**（我和你差 3 个位置） |
| 额外参数量 | max_seq_len × d_model = 49,152 | 0 |
| 能否泛化到更长序列 | **不行**，表没有的行查不了 | 可以，配合位置插值效果好 |

这也正是 Nova 后续要做的改造方向——从可学习绝对位置嵌入切换到 RoPE，彻底打破上下文窗口的硬上限。
